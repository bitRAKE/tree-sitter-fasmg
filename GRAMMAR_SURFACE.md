# tree-sitter-fasmg — Grammar surface

Reference for anyone writing a tool on top of `tree-sitter-fasmg` —
syntax highlighter, linter, language server, formatter, navigator.
This document is the *contract*: what the grammar emits, what fields
to read, and what invariants hold.

Status: **grammar phases 1–10 complete; 316/321 corpus baseline
(98.44 %) as of commit 15431ce.** The five remaining failures are
listed under [Known limitations](#known-limitations).

Related:

- [grammar.js](grammar.js) — the source of truth.
- [design-notes/edge-cases-plan.md](design-notes/edge-cases-plan.md)
  — historical context for phases 7–10 (free clauses, stray brackets,
  `calm_target`).
- [design-notes/refactor-plan.md](design-notes/refactor-plan.md) —
  the earlier structural refactor (phases 1–6).
- [src/node-types.json](src/node-types.json) — machine-readable list
  of every node kind + field emitted (regenerated by
  `tree-sitter generate`).

## 1. Philosophy

Three principles dictate the shape of the tree:

1. **Position-based classification, not keyword-based.** Whether a
   word is an instruction, expression, or labeled instruction depends
   on *where it sits in the line*, not on which keyword it matches.
   The grammar reflects this by routing everything through
   `_statement_head` and letting the per-statement rules (`labeled_statement`,
   `instruction_statement`, etc.) decide classification — no
   lexer-level keyword precedence above identifiers.
2. **Accept what fasmg accepts.** fasmg is a macro assembler; it
   routinely accepts constructs that look malformed in isolation but
   resolve after macro expansion. The grammar's
   `permissive_block_body`, `free_begin_clause`, `free_end_clause`,
   and `stray_bracket` rules are all there to let the grammar parse
   bodies that are syntactically incomplete on their own but
   legitimate fasmg.
3. **Fields are the contract; node kinds are the substrate.** Tools
   reading the tree should prefer named fields
   (`node.child_by_field_name("name")`) over positional indexing
   (`node.child(1)`). Field names are stable across grammar
   revisions; positional order is not.

## 2. Top-level shape

```
source_file
├── _statement (repeated)
│   ├── _statement_head
│   │   ├── line_modifier?            ← field "line_modifier"
│   │   └── <one definition | block | simple statement>
│   └── \n
└── _statement_no_nl?                 ← final line without trailing \n
```

`source_file` is the root. It alternates `_statement` (which ends in
`\n`) with `_blank_line` (a run of `\n`+). The last line may omit its
trailing newline via `_statement_no_nl`.

`_statement_head` is a *hidden* node (leading `_`) — it doesn't show
in the tree. Its children appear as direct children of `_statement`
instead, so the effective shape of a statement child is "optional
line_modifier field, then the statement body."

## 3. Statement families

Every statement is one of three families:

### 3.1 Definition statements

Open a body that runs until an explicit `end` phrase.

| Node kind | Keyword | Body rule | End phrase |
|---|---|---|---|
| `macro_definition` | `macro` | `block_body` (aliased `permissive_block_body`) | `end macro` |
| `struc_definition` | `struc` | `block_body` (aliased `permissive_block_body`) | `end struc` |
| `calminstruction_definition` | `calminstruction` | `calm_block_body` | `end calminstruction` |

All three carry the same fields:

| Field | Optionality | Content |
|---|---|---|
| `name` | required | `symbol_name` of the definer. |
| `decorator` | repeated, both before and after `parameters` | `":"` or `"!"` tokens. Order is the tree's, preserve it. |
| `parameters` | optional | `parameter_list` of comma-separated `parameter` nodes. |
| `body` | optional | Block body (one of the rules in the table). Absent when the definer has no body lines. |
| `end` | required | The literal `end X` phrase — always emitted, even if the body is empty. |
| `target` | `struc_definition` / `calminstruction_definition` only, optional | `calm_target` node: `( &IDENT )` wrapping the binding's case-insensitive name. |

### 3.2 Block statements

Open a paired block that runs until a specific `end X` phrase. Body
is always `block_body` unless noted. Most carry a `condition`,
`arguments`, or `pattern` field.

| Node kind | Opener | End | Primary field |
|---|---|---|---|
| `namespace_block` | `namespace` | `end namespace` | `arguments` (optional) |
| `if_block` | `if` | `end if` | `condition` (required); plus repeated `else_if_clauses` and optional `else_clause` |
| `while_block` | `while` | `end while` | `condition` (optional) |
| `repeat_block` | `repeat` / `rept` | `end repeat` | `arguments` (optional) |
| `iterate_block` | `iterate` | `end iterate` | `arguments` (optional) |
| `irp_block` | `irp` / `irpv` | `end irp` / `end irpv` | `arguments` (optional) |
| `match_block` | `match` / `rawmatch` / `rmatch` | `end match` | `pattern` (optional); plus repeated `else_match_clauses` and optional `else_clause` |
| `virtual_block` | `virtual` | `end virtual` | `arguments` (optional) |
| `postpone_block` | `postpone` | `end postpone` | `arguments` (optional) |

Every block node has a `body` field (the `block_body`) and an `end`
field (the closing phrase). `if_block` and `match_block` additionally
emit `else_if_clauses` (repeated) / `else_match_clauses` (repeated)
/ `else_clause` (optional) fields.

### 3.3 Simple statements

Single-line constructs that don't open a body.

| Node kind | Role | Fields |
|---|---|---|
| `labeled_statement` | one or more `label_definition`s followed by an optional simple statement | no explicit fields; children are `label_definition` (one or more) + `_simple_statement` |
| `assignment_statement` | `= / := / =: / equ / reequ / define / redefine` | `name`, `operator`, `value` (optional) |
| `directive_statement` | fasmg control directive (`include`, `display`, etc.) | `keyword`, `arguments` (optional) |
| `instruction_statement` | assembly instruction or macro call | `name`, `arguments` (optional, of type `instruction_argument_list` or `comma_prefixed_argument_list`) |
| `restore_statement` | `restore` | `names` (a `name_list`) |
| `purge_statement` | `purge` | `names` (a `name_list`) |

## 4. CALM sub-language

A `calminstruction_definition` body is a `calm_block_body` — a
different grammar sub-language from the top-level. Consumers who
care about CALM semantics (theming, semantic tokens, navigation)
should dispatch on the enclosing node kind.

`calm_block_body` alternates `calm_statement` with `_blank_line`.
Each `calm_statement` is one of:

| Node kind | What it wraps |
|---|---|
| `labeled_calm_statement` | `label_definition` + optional `calm_simple_statement` |
| `calm_simple_statement` | `asm_statement` / `match_statement` / `restore_statement` / `purge_statement` / `assignment_statement` / `directive_statement` / `instruction_statement` |

Inside CALM:

- `match` is **single-line** (`match_statement`), not a block.
- `asm` emits whatever follows as a top-level-style sub-statement —
  routed through `asm_statement { keyword: "asm", statement: (…) }`.
- Label definitions follow the top-level `label_definition` rule but
  are semantically `calmLocalLabel` rather than `globalLabel`.

Consumers that want to theme CALM constructs separately (e.g.
`calmParam`, `calmDirective`) should classify by ancestor: any
descendant of `calm_block_body` is in CALM scope.

## 5. Arguments and expressions

### 5.1 Lists

| Node kind | Shape |
|---|---|
| `argument_list` | `argument` (`,` `argument`)* `,`? |
| `instruction_argument_list` | `instruction_argument` (`,` `argument`?)* — first atom is special-case |
| `comma_prefixed_argument_list` | `,` `argument_list` — only used for instruction statements starting with `,` |
| `parameter_list` | `parameter` (`,` `parameter`)* `,`? |
| `name_list` | `symbol_name` (`,` `symbol_name`)* `,`? |

### 5.2 Argument content

| Node kind | Role |
|---|---|
| `argument` | Either an `angled_argument` or one-or-more `argument_atom`. |
| `argument_atom` | A token of the argument's expression. See below. |
| `instruction_argument` | Either an `angled_argument` or `instruction_argument_head` followed by zero-or-more `argument_atom`. |
| `instruction_argument_head` | First atom; distinguishes from tail atoms because no `stray_bracket` allowed here. |

**`argument_atom` choices** (in grammar order):

- `number_literal`, `string_literal`, `built_in_symbol`
- `sigil_token` (`=name`, `:name`, `` `name ``, `=:name`)
- `word_operator` (`and`, `or`, `xor`, `not`, `eq`, …)
- `identifier`, `operator_token`
- `parenthesized_argument`, `bracketed_argument`, `braced_argument`
- `stray_bracket` — a bare `]`, `)`, or `}` (see §7.3)

**Paired bracket nodes**: `parenthesized_argument`,
`bracketed_argument`, `braced_argument` each carry an optional
`argument_list` inside. They are distinct node kinds; consumers
can theme or fold them independently.

`angled_argument`: `<` content `>` where content is
`angled_text_fragment` + `line_continuation`. Used for inline macro
text.

### 5.3 Parameters

`parameter`:

```
(parameter
  "&"?                                          ← optional leading &
  name: (symbol_name)
  ("?" | "*" | "&")*                            ← name-level decorators
  (":" default: (argument))?                    ← default value
)
```

Parameter decorators are **not a field** (they're raw token
children); consumers inspecting them should walk the children
between `name` and `default`. The name-level `?` suffix means
"catch-all in this class"; `*` means "tail-all"; `&` means
"receive raw argument". A parameter may carry more than one
(e.g. `?name&`).

### 5.4 Expressions

`_expression` is a hidden node used inside specific contexts. It's
a limited subset: `number_literal`, `string_literal`, `identifier`,
and `(` `_expression` `)`. Most expression-like content in the
grammar uses `argument`/`argument_atom` instead — the generic
path.

## 6. Line modifiers and decorators

### 6.1 Line modifiers

```
_statement_head
└── line_modifier                     ← field "line_modifier"
    └── "{" argument_list? "}"
```

`{i286}`, `{386}`, `{x86_64}`, etc. — a bare `{...}` at the start of
a statement. The grammar emits a `line_modifier` node whose content
is an optional `argument_list`. Consumers that care about
architecture gating read this field and flatten the argument list to
a string.

### 6.2 Name decorators

Definer names (`macro_definition.name`, `struc_definition.name`,
`calminstruction_definition.name`) can carry repeated `decorator`
fields with values `":"` or `"!"`. Order matters — it's the tree's
order — and decorators can appear *both before and after* the
`parameters` field:

```fasmg
macro foo! : params, go :: body
```

The tree:

```
(macro_definition
  name: (symbol_name "foo")
  decorator: "!"
  decorator: ":"
  parameters: (parameter_list …)
  decorator: ":"
  decorator: ":"
  body: (block_body …)
  end: …)
```

Consumers should preserve decorator order when rendering
(hover tooltips, signature-help, documentation).

### 6.3 Optional `!` prefix on definer heads

`macro!`, `struc!`, `calminstruction!` — the `!` is an unconditional-
definition marker. The grammar emits it as a bare `"!"` child of the
definition node (not inside `decorator`), between the keyword and
the `name` field. It's the "esc" / "!" form from the fasmg manual.

## 7. Edge-case rules

### 7.1 `free_begin_clause`

```
(free_begin_clause
  <one of:
     namespace arguments?
     if condition
     while condition?
     iterate arguments?
     repeat arguments?
     match pattern
     virtual arguments?
     postpone arguments?
     irp arguments? | irpv arguments?
  >
  "\n")
```

A block opener line that's *not* paired with its own closer because
the closer lives in a peer macro body (classic `struct?`/
`end?.struct?!` pattern). Lives inside `permissive_block_body` —
which is aliased to `block_body` for the definition-rule consumers,
so the surface tree shows `body: (block_body …)`.

Declared at `prec.dynamic(-1)`, so the full block form wins when it
can close. Consumers that navigate block structure should treat
`free_begin_clause` as "opener without body" — it doesn't descend.

### 7.2 `free_end_clause`

External-scanner token (no structured children). Emitted when the
scanner sees `end X` where `X` is a block-closer keyword (`namespace`,
`if`, `while`, `iterate`, `repeat`, `match`, `virtual`, `postpone`,
`irp`, `irpv`, `data`) in a `permissive_block_body`. The scanner
*refuses* to emit `free_end_clause` for `end macro` / `end struc` /
`end calminstruction` because those are the definition-rule
closers — the structured grammar's own `end` field has to see them.

### 7.3 `stray_bracket`

A bare `]`, `)`, or `}` standing alone as an `argument_atom`. fasmg's
syntactical characters terminate tokens but don't require balancing
at parse time, so `dtail]` is a legitimate sequence of
`identifier "dtail"` + `stray_bracket "]"`. Declared at
`prec.dynamic(-10)` so structured pairs win when both halves exist
in the same argument. **Stray *openers*** (`[`, `(`, `{`) are **not**
recovered — trying to include them regressed the corpus by 41 files
(see [design-notes/edge-cases-plan.md §Phase 9](design-notes/edge-cases-plan.md)).

### 7.4 `match_pattern_text`

External-scanner token. A `match_pattern` is either a structured
`argument_list` or an opaque `match_pattern_text`. The scanner
decides — when it sees a pattern containing an unclosed opener at
EOL (a regular occurrence in fasmg match patterns), it produces
`match_pattern_text` with the raw line text rather than trying to
force a bracket balance. Consumers that want to analyse the pattern
contents must re-parse `match_pattern_text` against fasmg's pattern
grammar (not modelled here).

### 7.5 `calm_target`

Field on `struc_definition` and `calminstruction_definition`,
optional. Shape: `( &? IDENT )`. The inner identifier is the
case-insensitive name the definer binds into (e.g.
`(&fword)`). Consumers doing name resolution should prefer
`calm_target.child_by_field_name(nothing — walk children)` and
pick the `identifier` child.

## 8. Tokens

Most are conventional; the unusual ones:

| Token | Pattern | Notes |
|---|---|---|
| `identifier` | `/[^\s+\-/*=<>()\[\]{}:!,|&~`;\\#]+/` at `prec(-1)` | Very permissive; used as the fallback. |
| `placeholder_name` | `/\?+/` | `?`, `??`, `???` — anonymous placeholders. |
| `symbol_name` | `(identifier | placeholder_name)` (`#` `identifier | placeholder_name`)* | The `#` is fasmg's name-concatenation operator (static). |
| `number_literal` | nine patterns at `prec(2)` (hex with `0x`, `$`, `h`; binary with `b`; octal with `o`/`q`; decimal with optional `e` exponent, `f` suffix, `d` suffix; float) | Exhaustive list lives in `grammar.js:32-42`. |
| `string_literal` | `'...'` with `''` escape, or `"..."` with `""` escape, at `prec(2)` | No backslash escapes — that's fasmg semantics. |
| `sigil_token` | Four patterns at `prec(3)`: `=:name`, `=name`, `` `name ``, `:name` | Name-producing operators (deref, ref, etc.). |
| `built_in_symbol` | list-driven from `spec/keyword-groups.json` at `prec(3)` | Built-in names (registers, encoding keywords) that always win over plain identifiers. |
| `operator_token` | single-char or two-char ops at `prec(-1)` | Low precedence so keywords can beat them. |
| `word_operator` | CI-matched word operators (`and`/`or`/`xor`/`not`/`eq`/…) | List driven from `spec/keyword-groups.json`. |
| `directive_keyword` | CI-matched directive names (`include`/`display`/`err`/…) | List driven from `spec/keyword-groups.json`. |
| `comment` | `;` to EOL | In `extras`, so can appear between any two other tokens. |
| `line_continuation` | `\` + `\n` | Also `extras`. |

Case-insensitive matching for keywords is via the `ciWord` /
`ciPhrase` / `keywordToken` helpers at the top of `grammar.js`.

## 9. Conflicts and precedence

`grammar.js` declares these conflicts (GLR tie-breakers):

- `[$.parameter]` — default resolution for the standard parameter
  rule.
- `[$.macro_definition]`, `[$.struc_definition]`,
  `[$.calminstruction_definition]` — keep the definition rules
  independent when they share a prefix.
- `[$.<block>_block, $.free_begin_clause]` for each of the nine
  opener keywords — lets GLR explore both full-block and opener-only
  paths, picking whichever can complete.

Key dynamic-precedence decisions:

- `free_begin_clause`: `prec.dynamic(-1)` — full block wins when it
  can; this falls through only when no closer is present.
- `stray_bracket`: `prec.dynamic(-10)` — structured pair always
  wins when present; stray only kicks in for bare closers in the
  wild.
- `instruction_statement`: `prec.right(-1)` — low priority so
  keyword-led rules can intercept when applicable.
- `_expression` and `argument`: right-associative via
  `prec.right`.

## 10. Queries

Three `.scm` files ship with the grammar, consumed via
`tree-sitter-fasmg::HIGHLIGHT_QUERY` / `LOCALS_QUERY` / `FOLDS_QUERY`
in [bindings/rust/lib.rs](bindings/rust/lib.rs).

### 10.1 `queries/highlights.scm`

Captures standard highlight classes for the grammar's main public
surface: `@comment`, `@string`, `@number`, `@attribute`
(`line_modifier`), `@property` (labels), `@function`
(`macro_definition` + `instruction_statement` names),
`@function.builtin` (`calminstruction_definition.name`), `@type`
(`struc_definition.name`), `@type.builtin` (`calm_target`
identifier), `@variable`, `@variable.parameter`, `@keyword`,
`@punctuation.special` (decorators), and `@punctuation.bracket`
(`stray_bracket`).

Recovery-only nodes (`free_begin_clause`, `free_end_clause`) are
deliberately highlighted as `@keyword` so browser/manual-diagnostic
consumers can spot them immediately.

### 10.2 `queries/locals.scm`

Captures: `@local.scope` on `macro_definition.name` and
`struc_definition.name` and `calminstruction_definition.name`;
`@local.definition` on `label_definition.name`,
`assignment_statement.name`, and `parameter.name`;
`@local.reference` on every `identifier`.

Still conservative — it doesn't distinguish CALM-local from top-level
scopes, so advanced shadowing analysis still belongs in a higher-level
consumer.

### 10.3 `queries/folds.scm`

Captures `@fold` on every major definition/block form emitted by the
grammar: `macro_definition`, `struc_definition`,
`calminstruction_definition`, `namespace_block`, `if_block`,
`while_block`, `iterate_block`, `repeat_block`, `match_block`,
`virtual_block`, `postpone_block`, and `irp_block`.

## 11. Invariants consumers should know

These aren't enforced by the grammar syntactically but are true in
practice; building a tool on wrong assumptions will produce spooky
bugs.

1. **`end X` is never optional.** Every definition and block has a
   mandatory `end` field. The `free_begin_clause`/`free_end_clause`
   rules exist to handle openers and closers that live in *different*
   macro bodies; they don't make `end` optional — they route it.
2. **Opener and closer can live in different macro bodies.** A
   `macro struct?` body may open `namespace .` with no `end namespace`
   in the same body; the peer `macro end?.struct?!` body closes it.
   Consumers doing cross-body analysis must resolve this themselves
   (macro expansion is out of the grammar's scope).
3. **Classes are position-based.** `byte` in `byte 0` is an
   instruction; `byte` in `a: byte` is a labeled instruction; `byte`
   in `equ byte + 1` is an expression term. No lexer-level hint
   distinguishes them; only the enclosing statement rule does.
4. **CALM scope is ancestor-gated.** Descendants of `calm_block_body`
   are CALM — anywhere else isn't. There's no per-node "I'm CALM"
   marker.
5. **Decorators preserve order.** The `repeat(field("decorator", …))`
   in definer rules emits decorators in source order. Consumers that
   re-serialise (formatters, hover) must preserve it.
6. **`line_modifier` is per-statement.** Every `_statement_head` can
   carry one; it's not a file-level directive. Consumers doing
   architecture gating look at the `line_modifier` field on the
   specific statement, not at any ambient context.
7. **Parameter catch-all syntax is a repeated suffix, not a
   decorator.** `?name&` is parsed as `"?" "name" "&"` inside
   `parameter` — three tokens, not a name-with-decorator. Consumers
   classifying `?!`/`!?` catch-alls should examine the token sequence
   around the `name` field rather than looking for a single decorator
   atom.

## 12. Building the grammar

The CLI is resolved via `$TREE_SITTER_PATH` (defaults to
`C:\third_party\tree-sitter`); see [README](README.md) for the shim
contract.

```bash
bash scripts/ts.sh generate --js-runtime native   # regenerate src/ from grammar.js
bash scripts/ts.sh test                           # run test/corpus/*
bash scripts/build-wasm.sh                       # build ./tree-sitter-fasmg.wasm for browser consumers
bash scripts/export-playground.sh out/           # export stock Tree-sitter playground as static files
```

PowerShell users: `.\scripts\ts.ps1 generate --js-runtime native` /
`.\scripts\ts.ps1 test` / `.\scripts\build-wasm.ps1` /
`.\scripts\export-playground.ps1 out\`.

Corpus baseline against external fasmg corpora:

```bash
bash scripts/corpus-baseline.sh        # refresh ./baseline/ in place
bash scripts/corpus-baseline.sh out/   # candidate run; prints delta vs baseline/
```

Defaults to Grysztar's `fasmg/packages` + `fasm2/include`. Override
via `FASMG_CORPORA` (colon-separated MSYS paths). Outputs land in
`baseline/{results.csv,failures.tsv,summary.md,raw-summary.json}`;
`summary.md` records pass count, throughput (bytes/ms), and per-file
inner+outer ERROR positions for the regressors.

Consumers of the Rust bindings rebuild automatically — `build.rs`
invokes the C compiler against `src/parser.c` + any custom scanner
(we have one at `src/scanner.c`) and exposes the resulting symbol via
`LanguageFn::from_raw`.

Other bindings (Node, WASM, Swift, etc.) are not currently generated.
Add them under `bindings/<language>/` when a consumer needs them.

| `FOLDS_QUERY` | `&'static str` | Content of `queries/folds.scm`. |
| `STANDARD_HIGHLIGHT_NAMES` | `&'static [&'static str]` | Standard capture classes recognised by the Rust HTML helper. |
| `DEFAULT_HTML_CSS` | `&'static str` | Bundled stylesheet for `highlight_html_document`. |
| `highlight_html` | `fn(&str) -> Result<HtmlHighlightOutput, HighlightHtmlError>` | Render a fragment of class-based HTML plus syntax diagnostics. |
| `highlight_html_document` | `fn(&str, &str) -> Result<String, HighlightHtmlError>` | Wrap the highlighted fragment in a standalone HTML document. |
| `highlight_html_with_query` | `fn(&str, &str) -> Result<HtmlHighlightOutput, HighlightHtmlError>` | Same as `highlight_html`, but with a caller-supplied highlight query. |

`tree-sitter-highlight` crate and returns CSS-class HTML so downstream
sites can apply their own theme.

## 14. Known limitations

The 316/321 baseline is 5 files short. All five are documented
accepted failures. Consumers should expect `ERROR` nodes on:

| File | Snippet | Cause |
|---|---|---|
| `fasm2/macro/if.inc` 110:17 | `current equ (` | Stray open paren in equ RHS; `stray_bracket` doesn't include openers. |
| `fasmg/x86/include/macro/if.inc` 110:17 | same | fasmg copy of the above. |
| `fasmg/utility/inline.inc` 31:1 | `end struc` | `end struc` in peer body; can't distinguish from surrounding `struc_definition`'s own closer. |
| `fasm2/x86-2.inc` 75:2 | `end calminstruction` | Builder-builder pattern: `calminstruction` body emits `end calminstruction` into parent scope. Unrecoverable without macro-expansion context. |
| `fasmg/x86-2/x86-2.inc` 75:2 | same | fasmg copy of the above. |

See [design-notes/edge-cases-plan.md §Post-implementation status](design-notes/edge-cases-plan.md)
for the full analysis.

Tools should degrade gracefully on `ERROR` nodes — emit diagnostics
around them, don't crash. The error node's span points at the
outermost unrecoverable region; the baseline's innermost-error
reporting helps pinpoint the actual failing token.

## 15. Changelog surface (for downstream pinning)

When pinning a specific grammar commit, tools should record:

- **Commit SHA** of this repo.
- **Baseline percentage** at that SHA (see `baseline/summary.md`).
- **`node-types.json` hash** — if the grammar gains new node kinds,
  this hash changes and downstream walkers can detect the drift.
